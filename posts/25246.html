git 工作流程

  • 本地仓库——开发人员自己电脑上的Git仓库（.git）文件夹

一般流程

使用Git 与他人合作的一般工作流程可以概括为以下几个步骤：

  1. 克隆仓库 (Clone Repository):

    • 通过 git clone 命令从远程仓库中克隆代码到本地。

  2. 创建分支 (Create a Branch):

    • 在本地仓库中创建一个新的分支，用于开发新功能或修复 bug。

  3. 进行开发 (Development):

    • 在新创建的分支上进行代码的修改和开发。

  4. 提交更改 (Commit Changes):

    • 使用 git add 和 git commit 将修改的代码提交到本地仓库。

  5. 拉取更新 (Fetch and Pull):

    • 在提交更改之前，先拉取远程仓库的最新变更，确保本地代码与远程同步。

  6. 解决冲突 (Resolve Conflicts):

    • 如果在拉取更新时发生冲突，需要手动解决冲突。

  7. 推送更改 (Push Changes):

    • 将本地分支的更改推送到远程仓库。

  8. 创建合并请求 (Create a Pull Request):

    • 在远程仓库中创建一个合并请求（Pull Request，或简称 PR），请求将本地分支的更改合并到主分支或其他目标分支。

  9. 审查与合并 (Review and Merge):

    • 团队成员对合并请求进行审查，确认没有问题后，将其合并到目标分支。

  10. 删除本地分支 (Delete Local Branch):

    • 在合并后，可以删除本地分支。

  11. 删除远程分支 (Delete Remote Branch):

    • 如果需要，也可以删除远程分支。

什么是ssh密钥

SSH 密钥是 SSH（安全外壳）网络协议的访问凭证。这种经过身份验证和加密的安全网络协议用于不安全的开放网络上机器之间的远程通信。SSH 用于远程文件传输、网络管理和远程操作系统访问。 SSH 首字母缩略词还用于描述一组用于与 SSH 协议交互的工具。

SSH 使用一对密钥在远程方之间启动安全握手。密钥对包含公钥和私钥。私有与公共命名法可能会造成混淆，因为它们都被称为密钥。把公钥想成“锁”，把私钥想成“钥匙”会更有帮助。您向远程方提供公共“锁定”以加密或“锁定”数据。然后使用您保存在安全位置的“私钥”打开此数据。您可以使用 SSH（安全外壳协议）在 gitee 和 github 仓库上的存储库中访问和写入数据。通过 SSH 连接时，您使用本地计算机上的私钥文件进行身份验证。如需更多信息，请参阅“关于 SSH”。

简单来说，当你要登录到一个计算机或服务器时，你需要输入用户名和密码。这就像是你用一把钥匙打开房门一样。但是，有时候使用密码可能不够安全，因为密码可能会被猜测或者被别人知道。

SSH密钥就像是一把更安全的数字钥匙。它实际上是一对钥匙，一把是公开的（公钥），另一把是保密的（私钥）。公钥可以分享给任何人，但是只有持有相应私钥的人才能够成功连接。

当你想要连接到一台服务器时，你将自己的公钥分享给服务器。然后，当你尝试连接时，服务器会检查你的公钥是否与你的私钥匹配，如果匹配成功，你就被允许登录。

这样一来，即使别人知道你的公钥，他们也不能通过简单地猜测密码来访问你的账户。这使得连接更加安全和可靠。

远程仓库的作用

远程仓库在版本控制系统（Git）中起着重要的作用，其主要功能包括：

  1. 协作与分享： 远程仓库是团队协作的基石。团队中的每个成员都可以访问远程仓库，共享代码、文档和其他项目文件。这样，团队成员可以同时在一个项目上工作，而不会互相干扰。
  2. 备份： 将代码存储在远程仓库中有助于备份。本地仓库可能会丢失或损坏，但远程仓库可以充当主要代码存储的备份，确保项目数据的安全性。
  3. 版本管理： 远程仓库存储了项目的完整版本历史记录。这意味着你可以随时回退到先前的版本，比如修复错误或回到一个稳定的版本。
  4. 访问控制： 可以通过远程仓库进行权限和访问控制。只有授权的用户才能对仓库进行更改，这有助于管理团队成员之间的协作，并确保代码的安全性。
  5. 持续集成与部署： 远程仓库可以与持续集成和部署系统集成，使得每次提交都能够触发自动构建、测试和部署过程。这有助于确保代码质量和快速部署新功能。

选择.gitignore文件

.gitignore 文件是用来告诉 Git 哪些文件或文件夹不应该被纳入版本控制的配置文件。当你在项目中进行版本控制时，有些文件或目录可能是临时性的、机器生成的，或者包含敏感信息，这些文件通常不应该被纳入版本库，以免导致不必要的冲突或泄露敏感信息。

.gitignore 文件中列出的文件和目录模式将被 Git 忽略，不会被添加、提交或包含在版本库中。这样可以使版本库更加清晰、干净，同时避免不必要的文件进入版本控制。

以下是一些.gitignore文件中常见的模式：

  • 文件扩展名：*.log，表示忽略所有以 .log 结尾的文件。
  • 目录：/logs/，表示忽略根目录下的 logs 文件夹。
  • 通配符：build/，表示忽略所有名为 build 的文件或文件夹。
  • 注释：以 # 开头的行表示注释。

选择开源许可证

开源许可证是一种法律约定，规定了对于开源软件的使用、修改和分发的条件。这些许可证通过授予用户一定的权利，同时规定了用户需要满足的条件。开源许可证的主要目的是促进自由软件的开发和分享，同时确保原作者的权益。

以下是一些常见的开源许可证及其主要特征：

  1. MIT许可证： 允许在软件中使用、修改、合并、发布、销售副本，只要在软件和副本的所有副本或实质性部分中都包含原版权声明和许可声明。
  2. GNU通用公共许可证 (GPL): 强调用户对软件的自由使用、修改和分享，但有“传染性”，即基于GPL的代码的派生作品必须采用相同的GPL许可证。
  3. Apache许可证： 允许在软件中使用、修改、合并、发布，但必须包含原始版权、许可和免责声明。该许可证还提供了对贡献者的专利授权。
  4. BSD许可证： 允许在软件中使用、修改、合并、发布，要求在广告和软件文档中保留原始的版权声明。
  5. Creative Commons许可证： 用于非软件作品（如文档、图像、音乐等），提供了一系列许可选项，允许作者指定用户可以以何种方式使用他们的作品。

设置模板

readme

README 文件是一个包含在项目根目录下的文本文件，用于向其他开发者或用户提供关于项目的基本信息和使用说明。

通常，README 文件包含以下内容：

  1. 项目描述： 对项目的简要介绍，包括项目的名称、目的和主要功能。
  2. 安装说明： 描述如何安装项目的步骤，可能包括所需的依赖项、安装命令和配置信息。
  3. 使用说明： 提供项目的基本用法和示例，可能包括命令行示例、配置文件说明等。
  4. 贡献指南： 如果项目是开源的，可能包含对贡献者的一些建议、规则或指南。
  5. 许可证信息： 包含项目的许可证信息，明确了用户可以如何使用和分发项目。
  6. 项目结构： 如果项目有多个模块或组件，可能提供项目结构的简要说明。
  7. 联系方式： 提供项目维护者或团队的联系方式，以便其他人可以获取更多信息或报告问题。
  8. 版本历史： 记录项目的版本历史，包括每个版本的变更内容。
issue

在软件开发和项目管理中，”Issue”（问题）是指一个任务、缺陷、建议或其他需要关注的事项。在许多版本控制系统和协作平台中，”Issue”通常是指一个记录，用于跟踪和讨论项目的各种事务。(别人对你的代码指指点点的地方)

主要的特征和用途包括：

  1. 任务跟踪： Issue 通常用于追踪项目中需要完成的任务。每个 issue 都有一个标题、描述、分配给（负责人）、状态、标签等属性，使团队成员能够清晰地了解任务的当前状态和详细信息。
  2. 缺陷报告： 团队成员或用户可以使用 issue 来报告发现的软件缺陷或错误。这有助于维护团队对项目质量的认识，并及时解决问题。
  3. 功能建议： 项目的利益相关者可以使用 issue 来提出新功能的建议。这有助于团队了解用户需求，并计划未来的开发工作。
  4. 讨论和评论： 每个 issue 都可以附带讨论和评论，团队成员可以在其中交流观点、提供建议，或者提供解决问题的方法。
  5. 标签和里程碑： Issue 可以通过标签进行分类，例如指定优先级、类型、模块等。同时，它们可以与项目的里程碑（Milestone）相关联，以便更好地组织和计划工作。
  6. 版本控制集成： 在与版本控制系统集成的平台上，issue 通常与具体的代码提交和拉取请求（Pull Request）相关联，以便更好地追踪和理解与特定 issue 相关的代码变更。
pull request

Pull Request（PR）是一种在开源项目中进行代码合并的机制。它通常是在分布式版本控制系统（例如Git）中使用的术语，其中开发人员在完成他们的工作后，将其分支（branch）上的代码请求合并到项目的主分支（通常是主线或master分支）。

Pull Request 的基本流程如下：

  1. 分支创建： 开发人员从主分支上拉取（Pull）最新的代码，并在本地创建一个新的分支，用于进行特定的工作。
  2. 代码修改： 在新分支上进行代码的修改、添加新功能、修复缺陷等工作。
  3. 推送分支： 将修改后的分支推送（Push）到项目的远程仓库（通常是GitHub、GitLab等）。
  4. Pull Request 创建： 在远程仓库上创建一个 Pull Request，请求将新分支的修改合并到主分支上。
  5. 审查与讨论： 项目的维护者或其他贡献者可以在 Pull Request 上进行审查，提出建议、评论，进行讨论。
  6. 合并或关闭： 如果 Pull Request 经过审查没有问题，维护者可以选择合并（Merge）它到主分支。如果有问题或者需要进一步讨论，可以选择关闭 Pull Request。

选择分支类型

  1. 单分支模型 (只创建 master 分支)

在这个模型中，整个项目只有一个主分支，通常被称为 master。所有的开发、新功能、修复和发布都在这一个分支上进行。这是一种简单的模型，适用于小型项目或者对版本控制流程要求不太复杂的场景。但是，对于大型项目或需要同时处理多个版本的情况，这种模型可能显得不够灵活。

  1. 生产/开发模型 (支持 master/develop 类型分支)

这个模型引入了一个额外的 develop 分支。develop 分支用于集成所有开发者的工作，当 develop 上的代码达到一个稳定的状态时，将其合并到 master 分支，形成新的发布。这种模型使得开发过程更加灵活，允许开发者在 develop 分支上并行工作，而 master 分支则用于存放已发布的代码。

  1. 特性/发布模型 (支持 master/develop/feature 类型分支)

在这个模型中，除了 master 和 develop 分支之外，还引入了 feature 分支。每个新功能都会在一个独立的 feature 分支上进行开发，开发完成后再合并到 develop 分支。这有助于更好地组织和管理不同的功能开发。一旦 develop 分支上的功能达到某个发布的标准，就会被合并到 master 分支进行发布。

  1. 开发/发布分离模型 (支持 master/develop/feature/release 类型分支)

这个模型在生产/开发模型的基础上引入了 release 分支。release 分支用于准备新的发布，其中包括对版本号的增加、文档的更新和解决一些小的问题。当 release 分支准备好时，它会被合并到 master 和 develop 分支。这个模型使得可以在 release 分支上独立处理发布相关的事务，而不影响主要的开发过程。

  1. 开发/发布/缺陷分离模型 (支持 master/develop/feature/release/hotfix 类型分支)

这个模型在开发/发布分离模型的基础上引入了 hotfix 分支。hotfix 分支用于紧急修复生产环境中的问题，而不需要等待下一个预定的发布。修复完成后，hotfix 分支会合并到 master 和 develop 分支。这种模型允许在不影响正常开发流程的情况下，快速响应生产环境中的问题。

每个模型都有其优缺点，选择最合适的模型通常取决于项目的规模、团队的组织结构、开发流程和对版本控制的要求。较大和复杂的项目通常更适合使用包含多个分支类型的模型，而小型项目可能可以使用较为简单的模型。

git介绍

git 的基本功能

  1. 版本控制： Git 能够追踪文件的变化，记录每次修改的内容，以及谁在什么时候进行了修改。这使得开发团队能够轻松地查看和恢复历史版本，从而更好地管理代码。
  2. 分支管理： Git 允许创建和管理分支，这是代码独立开发的方式。每个分支都是项目的一个独立副本，开发者可以在不影响主干的情况下进行工作，最后再将修改合并到主分支。
  3. 协同合作： 多个开发者可以同时在同一个项目上工作，而不会干扰彼此的进度。Git 提供了远程仓库的概念，使得团队成员可以共享代码并协同开发。
  4. 快速提交： Git 提供了快速和轻量级的提交机制。每次提交都是一个快照，记录了整个项目在该时刻的状态，而不仅仅是变化的部分。
  5. 撤销更改： 如果出现错误或需要撤销某些更改，Git 提供了撤销更改的机制。这包括撤销工作目录中的修改、撤销已经提交的修改等。
  6. 远程仓库： Git 支持与远程仓库进行交互，允许开发者推送（push）和拉取（pull）代码。这为分布式开发提供了便利。
  7. 标签管理： Git 允许创建标签，这是对特定提交的命名引用。通常，标签用于标识发布版本，方便开发者和用户识别和使用。

git的工作原理

  1. 工作区（Working Directory）：
    • 这是你本地计算机上的实际项目目录，包含项目的所有文件和子目录。
    • 开发者在这里进行修改，添加、删除文件。
  2. 暂存区（Staging Area）：
    • 暂存区是一个中间区域，位于工作目录和本地仓库之间。
    • 在修改文件后，你需要将这些修改添加到暂存区，通过 git add 命令实现。添加到暂存区的文件即将被包含在下一次提交中。（idea 会自动的帮我把一些文件添加到暂存区中）
  3. 本地仓库（Local Repository）：
    • 本地仓库存储了项目的完整历史记录，包括所有的提交、分支、标签等信息。
    • 使用 git commit 命令将暂存区的内容提交到本地仓库。
  4. 分支（Branches）：
    • Git 使用分支来支持并行开发。默认情况下，项目有一个主分支（通常是 master）（一般我们不会直接在主分支上进行提交，我们一般是创建分支，然后在自己的分支提交并审查完之后才能跟主分支合并），但你可以创建新的分支并在不影响主分支的情况下进行开发。
    • 使用 git branch 和 git checkout 命令创建和切换分支。
  5. 远程仓库（Remote Repository）：
    • 远程仓库是项目在网络上的存储库，通常托管在服务如GitHub、GitLab或Bitbucket上。
    • 使用 git push 和 git pull 命令，你可以将本地仓库的修改推送到远程仓库或从远程仓库拉取更新。
