设计模式

UML 入门

  1. 关联关系：

    • 双向关联：客户与商品（我中有你，你中有我）

    • 单向关联：包含（我包含你）

    • 自关联：链表的结点

    • 多重性关联：表示关联对象在数量上的对应关系

    • 聚合关系：表示整体与部分的关系，但是部分可以脱离整体而存在（你是我的装备）

    • 组合关系：表示整体与部分的关系，但是整体对象可以控制成员对象的生命周期，同生共死（你是我的牙齿）

  2. 依赖关系（我会用到你）

    是一种使用关系，表示一个事物使用另一个事物，通常有三种实现方式

    • 直接把一个对象作为一个类中方法的参数
    • 在一个类的方法中将另一个对象作为其局部变量
    • 在一个类的方法中调用另一个类的静态方法

  3. 泛化关系（继承）

    继承关系，用于表示父类与子类之间的关系，弗雷有被称作基类或超累，子类又被称为派生类。

  4. 接口与实现关系（实现）

面向对象设计原则概述

大多数情况下，开闭原则是目标，里氏代换是基础，依赖倒转是手段，相辅相成

  1. 单一职责原则：一个类只负责一个功能领域中的相应职责，就一个类而言，应该只有一个引起他变化的原因

    理由：如果一个类承担的职责越多（链接数据库，查询信息，创建图标），那么它被复用的可能性就越小

  2. 开闭原则：一个软件实体应当对扩展开放，对修改关闭。软件实体应该尽量在不修改原有代码的情况下进行扩展

    理由：这是可复用设计的一块基石

    关键：抽象化，将系统定义为一个相对稳定的抽象层，将不同的实现行为移至具体的实现层中完成（定义系统的抽象层，再通过具体类来进行扩展，如果需要修改系统行为，无需对抽象层进行任何改动，只需要增加新的具体类来实现新的业务功能即可）

  3. 里氏代换原则：所有引用父类的地方，必须能够透明的使用其子类的对象

    理解：将一个父类对象替换成它的子类对象，程序将不会产生任何错误和异常，反过来则不成立

    理由：历史代换原则是实现开闭原则的重要方式之一（在程序中使用基类类型来对对象进行定义，在运行时再确定其子类类型，使用子类对象来替换父类对象，可以很方便的对系统进行升级）

    关键：将父类设计为抽象类或接口，让子类继承父类或实现父类接口，并实现在父类中声明的方法

  4. 依赖倒转原则：抽象不应该依赖于细节，细节应当依赖于抽象，这对接口编程，而不是针对实现编程

    理由：程序中使用抽象层进行编程，将具体类写在配置文件中，如果系统行为发生变化，只需要对抽象层进行扩展，并修改配置文件即可

    关键：

    • 在程序代码中传递参数或在关联关系中，尽量引用层次高的抽象层类
    • 依赖注入

    示例：

    增加新数据类型需要修改 CustomerDAO

    增加新数据类型只需修改配置文件

  5. 接口隔离原则：使用多个专门的接口，而不使用单一的总接口，客户端不应该依赖那些它不需要的接口

    示例：

    接口承担太多职责，倒是实现类庞大，不同实现类中不得不实现接口中定义的所有方法，灵活性差

  6. 合成复用原则 ：尽量使用对象组合，而不是继承来达到复用的目的

    理由：可以降低类之间的耦合度，一个类变化对其他类造成的影响较少，如果要继承，严格遵循里氏代换

    关键：尽量使用关联关系，少用继承

    PS：合适用复用，何时用继承，has 用复用，is 用继承

    示例：

    如果此时我们要更改数据库，则要修改两个类中的其中一个，违反开闭原则

    新的设计方式中，如果要更改数据库，则可以直接使用依赖注入的方式

  7. 迪米特法则：一个软件实体应当尽可能地减少与其他实体发生相互作用

    理由：如果符合此法则，当其中一个模块发生修改时，就会尽量少地影响其他模块，扩展会相对容易，这是对软件实体之间通信的限制

    关键：尽量创建松耦合的类

    示例：

    在这个界面中，增加新的界面控件需要修改与之交互的其他空间的源代码，导致扩展性差

    引入一个专门控制界面空间交互的中间类，降低界面之间的耦合度，将请求转发给中间类，再又中间类来完成对其他控件的调用

创建型模式

创建型模式将对象的创建和使用分离，在使用对象时无需关心对象的创建细节，从而降低系统的耦合度。关键：创建什么，由谁创建，何时创建

Singleton

确保一个类只有一个实例，并且自行实例化并向系统提供这个实例，这个类成为单例类，提供全局访问方法

这种模式涉及到一个单一的类，该类负责创建自己的对象同时确保只有单个对象被创建。这个类提供了一种访问其唯一对象的机制，可以直接访问，不需要实例化该类的对象

介绍

动机：保证一个类仅有一个实例，并提供一个访问它的全局访问点

  • 确保对象只有一个实例
  • 想在程序上表现出“只存在一个实例”

主要解决：一个全局使用的类频繁的创建与销毁

何时使用：

  • 系统只需要一个实例对象，例如系统要求提供一个唯一序列号生成器或资源管理器，或者需要考虑资源消耗太大而只允许创建一个对象
  • 客户调用类的单个实例只允许使用一个公共访问点，除了该公共访问点，不能通过其他途径访问该实例

如何解决：判断系统是否已经有这个单例，如果有则返回，如果没有则创建。

关键代码：私有构造函数，设置私有静态成员变量，增加共有静态访问方法

应用实例：

  • Windows 是多进程多线程的，在操作一个文件的时候，不可避免的会出现多个进程或线程同时操作一个文件的现象，所以所有文件的处理必须通过唯一的实例来进行

  • 一些设备管理器常常设计为单例模式，比如一个电脑有两台打印机，在输出的时候就要处理不能两台打印机打印同一个文件

  • 负载均衡器的设计

  • 生产唯一序列号，计数器等

优点：

  • 在内存里只有一个实例，减少了内存的开销，避免频繁的创建和销毁实例
  • 提供了对唯一实例的受控访问，封装了唯一实例，可以严格控制客户怎样以及何时访问
  • 可以通过扩展，让单例 变为 n 例

缺点：

  • 没有抽象层，扩展困难
  • 职责过重，违背单一职责原则。既提供了业务方法，又提供了创建对象的方法，将对象的创建和本身的功能进行耦合
  • 由于自动垃圾回收技术，实例化共享对象长时间不被利用，会被系统认为是垃圾，会自动销毁并回收资源，会导致共享单例对象状态丢失

注意事项：

需要注意多线程问题，记得加同步锁

假设一个单例对象还没有被创建，且创建需要消耗大量时间，但同时又两个请求要访问这个单例对象，那么这个时候有可能会创建出两个对象，违背单例模式的初衷

实现

饿汉式，线程安全，在类加载的时候就已经创建了单例对象，缺点是不能在系统需要的时候才创建这个单例类，可能会造成资源浪费

懒汉式，延迟加载，使用 synchronized 关键字修饰 get 方法，避免多个线程同时调用 get 方法，缺点，每次调用该方法的时候都需要进行线程判定，会导致系统性能降低，不适用于多线程高并发访问环境

改进：锁定临界资源块，问题依然存在，如果两个线程同时调用 get 方法，都能通过判断，导致依然可能产生多个单例对象

改进：使用双重检查锁定（检查，锁定，检查），需要在成员变量中添加 instance 修饰符，确保多个线程都能正确处理，会屏蔽虚拟机所做的一些代码优化

饿汉式单例类不能实现延迟加载，始终占据内存，懒汉式单例类线程安全控制繁琐，且受性能影响

Initialization on Demand Holder(IoDH)，将成员变量改为 静态内部类 在内部类中创建单例对象，再将该单例对象通过 get 方法返回给外部使用

单例对象没有作为成员变量直接实例化，因此类加载时不会实例化

Simple Factory

定义一个工厂类，可以根据参数的不同方位不同的类实例，被创建的实例通常都具有共同的父类。因为简单工厂模式中用于创建实例的方法是静态的方法，因此简单工厂模式又被称为静态工厂方法模式

介绍

动机：将类的创建和使用进行解耦

主要解决：单一类职责过重，创建和使用都在一个类中

何时使用：

  • 工厂类负责创建的对象比较少，不会造成工厂方法业务逻辑负责
  • 客户端只知道传入工厂类的参数，对于如何创建对象并不关心

如何解决：工厂（核心，根据传入的参数创建不同的产品），抽象产品（提供所有产品的公共业务方法和抽象业务声明方法），具体产品（实现抽象产品）

关键代码：工厂类提供静态方法，该方法根据传入的参数来创建不同的产品

优点：

  • 工厂类包含必要判断逻辑，可以决定什么时候创建哪个产品类的实例，客户端可以免除直接创建产品对象的职责，而仅仅只是消费产品（客户端只负责消费），实现类对象创建和使用的分离
  • 客户端无需知道创建具体产品的类名，只需要知道具体产品所对应的参数即可
  • 通过引入配置文件，可以在不修改任何客户端代码的情况下更换和增加新的具体产品类，提高系统灵活性

缺点：

  • 工厂类集中了所有产品的创建逻辑，职责过重，一旦不能正常工作，整个系统都将受到影响
  • 使用简单工厂模式会增加系统中类的个数，增加系统的复杂度和理解难度
  • 系统扩展困难，一旦添加新产品就得修改工厂逻辑，不利于系统的扩展和维护
  • 使用静态工厂方法，工厂角色无法形成基于继承的等级结构

注意事项：

  • 不适用于存在大量产品的情况

Factory Method

定义一个用于创建对象的接口，让子类决定将哪一个类实例化

介绍

动机：在静态工厂模式中，静态工厂方法根据传入参数的不同来创建不同的产品，如果增加新产品，那么就一定会修改原代码

主要解决：简单工厂方法优化，解决工厂类职责太重的问题

何时使用：

  • 不想让客户端知道所需对象的具体类
  • 需要一个容易扩展的系统可以考虑使用

如何解决：抽象产品，具体产品，抽象工厂，具体工厂

关键代码：在简单工厂模式的基础上，抽象工厂，让具体工厂生产具体产品

应用实例：

  • 日志记录器，根据不同的文件来源创建不同的日志记录器
  • 各种开源框架和 API 类库的核心模式

优点：

  • 使用具体工厂方法创建需要的产品，向客户隐藏了哪种具体产品将被实例化这一细节，客户只需要关心所需产品的对应工厂，而无须关心创建细节，甚至不需要知道具体产品类的类名
  • 系统加入新产品时，无需修改抽象工厂和抽象产品提供的接口，无须修改客户端，无需修改其他具体类和具体产品，只需添加一个具体工厂和具体产品就行，完全符合开闭原则

缺点：

  • 增加新产品时，要添加具体工厂和对应的具体产品，系统中类的个数将成对增加
  • 引入抽象层，在客户端代码中均使用抽象层进行定义，增加系统的抽象性和理解难度，实现的时候可能需要用到 DOM ，反射，等技术，增加实现难度

注意事项：每个工厂只生产一类产品，可以考虑将相关产品组成一个产品族，由同一个工厂进行生产（抽象工厂模式）

Abstract Factiory

提供一个创建一系列相关或相互依赖对象的接口，而无需指定他们的具体类。

在工厂方法模式中，由于一种产品由一个工厂生产，这个时候，如果增加新的产品就会导致代码量急速增加，将一些相关产品组成一个产品族，由一个工厂统一生产，会更加便于系统扩展

产品等级结构

介绍

动机：工厂方法模式面对的是只有一个产品的产品等级结构，而抽象工厂模式面对的是具有多个产品的等级结构

主要解决：为创建一组对象提供解决方案

何时使用：当一个产品族中存在多种产品时，可以使用抽象工厂模式创建一个产品族

如何解决：抽象工厂，具体工厂（生产一组具体产品），抽象产品，具体产品

关键代码：抽象工厂

应用实例：

  • 界面控制器
  • 游戏根据不同操作系统提供不同控制类

优点：

  • 添加新的产品族很简单

缺点：

  • 添加新的产品等级结构很麻烦，需要修改大量代码，包括抽象层的代码

注意事项：需要提考虑之后的业务是否需要增加产品等级结构

实现

工厂方法模式实现一个界面皮肤更换系统

增加新皮肤时，虽然不需要修改现有代码，但是需要增加大量的类，导致系统庞大，增加系统的维护成本和运行开销

Prototype

使用原型实例来创建对象的种类，通过克隆一个原型实例来创建对象

介绍

动机：简化相同对象的创建

主要解决：创建一个对象成本很高，但又需要创建多个这样的相似对象，可以使用原型模式简化对象的创建过程

何时使用：

  • 创建新对象的成本较大（初始化占用时间长，占太多的CPU资源或网络资源）新对象可以通过原型模式对已有对象来复制获得，如果是相似对象，则可以对成员变量稍作修改
  • 如果系统要保存对象的状态，而对象的状态变化很小，或者对象本身占用的内存较少时，可以使用原型模式配合备忘录模式
  • 需要避免使用分层次的工厂类来创建分层次的对象，且类的实例对象只有一个或很少的几个组合状态，通过复制原型对象得到新实例可能比使用构造函数创建一个新实例更方便

如何解决：抽象原型类（声明克隆方法的接口），具体原型类（实现抽象原型类中声明的克隆方法，在克隆方法中返回自己的一个克隆对象）

关键代码：实现克隆接口

应用实例：

优点：

  • 简化对象创建过程，通过复制一个已有实例可以提高实例的创建效率
  • 扩展性好，在原型模式中提供了抽象原型类，在客户端可以针对抽象原型类进行编程，将具体原型类写在配置文件中，增加或减少对系统没有任何影响
  • 提供简化的创建接口，无需专门的工厂类来创建产品
  • 可以使用深克隆的方式保存对象的状态。

缺点：

  • 需要为每一个类配备一个克隆方法，且克隆方法位于一个类的内部，当要对该类进行修改的时候，需要修改源代码，违反开闭原则
  • 在实现深克隆的时候需要编写大量复杂的代码

注意事项：

实现

原型管理器：将多个原型对象储存在一个集合中供客户端使用，是一个专门负责克隆对象的工厂，原型管理器可以使用单例模式进行创建

Builder

介绍

动机：将对象的创建进行隔离

主要解决：复杂对象的组装与创建

何时使用：

  • 需要生成的产品对象有复杂的内部结构，这些产品对象通常包含多个成员变量
  • 需要生成的产品对象属性相互依赖，需要指定其生成顺序
  • 隔离复杂对象的创建和使用
  • 需要让对象的创建过程独立于创建该对象（创建对象的方法封装在指挥者类中）

如何解决：

  • 抽象建造者，（声明两类方法，一种是建造产品部件的方法，一种是直接返回具体产品的方法）
  • 具体建造者（实现抽象建造者），
  • 产品角色（被构建的复杂对象，包含多个组成部件，具体建造者创建产品内部表示并定义其装配过程），
  • 指挥者（安排复杂对象的建造次序，指挥者与抽象建造者之间存在关联关系，可以在其 construct 建造方法中调用建造者对象的不叫构造与装配方法，完成复杂对象的建造，客户端一般只需要与指挥者进行交互），隔离客户与创建过程，针对抽象构造者编程

关键代码：指挥者和建造者的编写（类似抽象工厂模式，不过抽象工厂模式生成的是一个产品等级，一组产品，建造者模式生成的是一个由多个产品组成的复杂产品）

应用实例：

优点：

  • 客户端不必知道产品内部的组成细节，将创建产品与产品本身进行结构使得相同的创建过程可以创建不同产品
  • 每一个具体建造者都相对独立，而与其他具体建造者无关，可以很方便地替换具体建造者或增加具体建造者
  • 增加指挥者类针对抽象建造者编程，使得增加新的建造者无须修改原有代码，扩展方便，符合开闭原则
  • 更加精细的控制产品的创建过程，将复杂产品的创建分解在不同的方法中，使得创建过程更加清晰

缺点：

  • 创建的产品一般具有较多共同点，组成部分相似，如果差异性很大，不适用（适用范围有限制）

  • 客户端可以一致的使用一个组合结构或其中的单个对象，不必关系处理的是单个对象还是整个组合结构

介绍

意图：提供一种方法顺序访问一个聚合对象中的各个元素，而无需暴露该对象的内部表示，是一种行为模式

主要解决：不同的方式来遍历整个整合对象（集合）

  2. 迭代器简化了聚合类。
  3. 在同一个聚合上可以有多个遍历
  4. 在迭代器模式中，增加新的聚合类和迭代器类都很方便而无需修改原有代码

缺点：

  1. 由于迭代器模式将储存数据和遍历数据的职责分离，增加新的聚合类需要增加新的迭代器类，类的个数成对增加，在一定程度上会增加系统复杂性

  • ConcreteIterator（具体的迭代器）

    负责实现 Iterator 所定义的接口

  • Aggregate（集合）

    负责定义创建 Iterator 角色的接口。这个接口是一个方法，会创建出”按顺序访问保存在我内部元素的人“

  • ConcreteAggregate（具体集合）

    负责实现 Aggregate 所定义的接口。能创建出具体的 Iterator 角色

在我们使用这个模式之后，可以将遍历与实现进行分离，如果我们放弃使用数组来对图书进行管理，不管BookShelf 如何变化，只要BookShelf方法能正确的返回 Iterator （这也是为什么 Iterator 方法的返回值不是BookShelfIterator的原因）的实例，那么我们无需对下面的遍历方法进行任何修改，代码都可以正常工作。

这种设计模式能极大的方便调用者，当程序的一个组件发生改变时，不需要对其他的组件进行修改或者只需要很小的修改即可应对，实现代码的可复用性

Adapter模式

介绍

意图：将一个类的接口转换成用户希望的另一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作

主要解决：如何将一些现存的对象放到新环境中，而新环境的接口是现对象不能满足（没有，不存在）的

何时使用：

  1. 系统需要使用现有的类，而此类的接口不符合系统需要。
  2. 想要建立一个可以重复使用的类，用于一些彼此之间没有太大关联的一些类，包括一些可能在将来引进的类一起工作，这些类源不一定有一致的接口。
  3. 通过接口转换，将一个类插入另一个类系中
  4. 接口升级：当我们需要修改一个接口，但又不希望影响到已有的客户端代码时，可以使用适配器模式来提供向后兼容的接口

如何解决：继承或依赖（推荐）

关键代码：如何对API进行包装，继承或依赖已有的对象，实现想要的目标接口

应用实例：Java中的JDBC，在LINUX上运行WINDOWS程序

优点：

  1. 可以让任何两个没有关联的类一起运行。
  2. 提高了类的复用性
  3. 增加了类的透明度
  4. 灵活性好

缺点：

  1. 过多使用适配器，会让系统非常凌乱，不容易整体进行把握。比如，命名看到调用的是A接口，其实内部被适配器改成了B接口的实现，一个系统如果太多出现这种情况，会是一场灾难
  2. Java最多继承一个类，所以最多只能适配一个适配者类，而且目标必须是抽象类

使用场景：有动机的修改一个正常运行的系统接口，这时应该考虑使用适配器模式。

注意事项：适配器不是在详细设计时添加的，而是解决正在服役的项目的问题

实现

  1. 使用继承

  2. 使用委托

    委托就是指将某个方法中的实际处理交给其他对象的方法

解析

  • Target（对象）

    该角色负责定义所需的方法，可以是接口，也可以是一个抽象类

  • Client（请求者）

    负责使用Target角色所定义的方法进行具体处理，负责使用的一方

  • Adaptee（被适配）

  • Adapter（适配）

    主角，负责对被适配者进行修改的地方，让被适配者能满足请求者的需求

通过这个模式，我们很容易根据已有的类去创建我们需要的方法群，当出现Bug时，由于我们明确的知道Bug不在现有的 类（被适配者）中，所以只需调查适配器即可

通常我们让现有的类适配新的接口时，常常会有只要将这里稍微修改下就可以了的想法，一不留神就会修改现有的代码，但是如果我们对已经测试完毕的现有代码进行修改，就必须在修改后重新进行测试。使用Adapter模式可以在完全不改变现有代码的前提下使用现有代码用于匹配新的API，此外如果没有现有代码，只要知道现有类的功能，就可以编写出新的类

Template 模式

在父类中定义处理流程的框架，在子类中实现具体处理

在模板模式中，一个抽象类公开定义了执行它的方法的方式/模板。他的子类可以按需要重写方法实现，但调用将以抽象类中定义的方式进行。是一种行为模式

介绍

意图：定义一个操作中算法的骨架，将一些步骤延迟到子类中。模板方法让子类可以不改变一个算法的结构即可重新定义该算法的某些特定步骤

主要解决：一些方法通用，却在每一个子类都重写了这一方法。

合时使用：有一些通用的方法

如何解决：将一些通用算法抽象出来

关键代码：抽象类，实现类（体会抽象类的作用）

应用实例：

  1. 奥特曼的每一集，就是一个顶层的逻辑骨架
  2. spring 中队 Hibernate 的支持，将一些已经定好的方法封装起来，比如开启事务，获取Session，关闭 Session等，程序员不重复写那些已经规范好的代码，直接丢一个实体就可以保存
  3. java.io.InputStream是一个抽象类，子类实现的抽象方法有 read()，availabe()，close()

优点：

  1. 封装不变部分，扩展可变部分
  2. 提取公共代码，便于维护
  3. 行为有父类控制，子类实现

缺点：

  1. 每一个不同的实现都需要一个子类来实现，导致类的个数增加，使系统更加庞大

使用场景：

  1. 多个子类共有的方法，且逻辑相同
  2. 重要的，复杂的方法，可以考虑作为模板方法

注意事项：为了防止恶意操作，一般模板方法都加上final关键词。

实现抽象类的子类

创建不同子类，调用抽象类中规定好的逻辑操作

解析

（这是一个为打工人设计的模式？大佬负责写顶层逻辑，小弟负责实现？？？）

在这个模式中，主要角色有两个

  • AbstractClass(抽象类)

    抽象类负责实现模板方法，还负责声明模板方法中所使用到的抽象方法。这些方法由子类负责实现

  • ConcreteClass(具体类)

    该角色负责实现抽象类中定义的抽象方法，这里实现的方法会在抽象类中的模板方法中被调用

在这个模式中，父类和子类联系紧密，共同工作，在子类中实现父类中声明的抽象方法时，必须理解这些抽象方法被调用的时机，在看不到父类的源代码的情况向，想编写出子类是非常困难的

在实例程序中，不管哪个子类都是保存在父类类型变量中，然后再调用逻辑方法的，使用父类变量保存子类实例的优点是，即使没有用 instanceof指定子类种类，程序也能正常工作，无论在父类类型变量中保存哪个子类的实例，程序都可以正常工作，这种原则被称为里氏替换原则（LSP）

扩展

在理解类的层次时，通常是站在子类的角度进行思考的，可以从以下方面入手

  • 在子类中可以使用父类中定义的方法
  • 可以通过在子类中增加方法以实现新的功能
  • 在子类中重写父类方法可以改变程序的行为

如果站在父类的角度进行思考，在父类中，我们声明了抽象方法，则希望达到以下目的

  • 期待子类去实现抽象方法
  • 要求子类去实现抽象方法

子类具有实现在父类中声明的抽象方法的责任。这种责任被称为子类责任

在抽象类阶段确定处理的流程非常重要

父类与子类的相互写作支撑起了整个程序，虽然将更多方法的实现放在父类中会让子类变得更轻松，但同时也降低了子类的灵活性；反之，子类则会变得臃肿不堪，还容易导致子类间的代码出现重复

java.io.InputStream使用了模板模式，需要子类实现的方法有read()，availabe()，close()

Factory 模式

在模板模式中，我们在父类中规定处理的流程，在子类中实现具体的处理。如果将该模式用于生成实例，那他就是抽象工厂模式，也就是将抽象的零件替换成抽象的产品

（抽象的概念 ：不考虑具体实现，只关心接口）

在工厂模式中父类决定实例的生成方式，但并不决定要生成的具体类，具体的处理全部交给子类负责。这样就可以将生成实例的框架（framework）和实际负责生成实例的类解耦

工厂模式提供一种将对象的实例化过程封装在工厂类中的方式，通过工厂模式，可以将对象的创建与使用代码分离，提供一种统一接口来创建不同类型的 对象

介绍

意图：定义一个创建对象的结构，让其子类自己决定实例化哪一个工厂类，工厂模式使其创建过程延迟到子类进行

主要解决：接口的选择问题

何时使用：我们明确的计划不同条件下创建不同实例时。

关键代码：抽象，封装

应用实例：

  1. JDBC连接工厂
  2. GUI组件工厂
  3. 日志记录器功能工厂
  4. 加密算法工厂

优点：

  1. 一个调用者想创建一个对象，只要知道名称就可以
  2. 扩展性高，如果增加一个产品，只要扩展一个工厂类就可以
  3. 屏蔽产品的具体实现，调用者只关心产品的接口

缺点：

  1. 每增加一个产品，就要增加一个具体类和对象实现工厂，使得系统中类的个数成倍增加，在一定程度上增加了系统的复杂度，同时也增加了系统具体类的依赖

使用场景：

  1. 创建复杂对象：当创建的对象过于复杂设计多个步骤或者依赖其他对象时，工厂模式可以将创建过程封装在工厂类中，使客户端无需关心实例化过程
  2. 统一管理对象的创建：如果有多出需要创建统一类型的对象，可以使用工厂模式将对象的创建逻辑集中在一个工厂类中，方便管理和维护
  3. 创建对象时涉及依赖注入：在依赖注入的场景中，工厂模式可以以作为一个创建依赖对象的方式，确保依赖对象的正确创建和管理

注意事项：

  1. 选择合适的工厂模式类型：
    • 简单工厂模式：用于创建一组相关对象，客户端只需知道产品的共同接口即可。
    • 工厂方法模式：用于创建单个产品类型的对象，每个产品类型有一个对应的工厂类。
    • 抽象工厂模式：用于创建多个相关对象的产品族，每个产品族有一个对应的工厂类。
  2. 适当的封装和抽象：工厂模式应该将对象的创建过程封装在工厂类中，并通过接口或抽象类来定义产品的公共接口。这样可以降低客户端代码与具体类的耦合。
  3. 使用依赖注入：在可能的情况下，可以将工厂对象作为依赖注入到客户端中。这样可以使客户端代码更加灵活，能够在运行时替换不同的工厂实现。
  4. 工厂方法的扩展性：如果预计需要添加新的产品类型，确保每个产品类型都有对应的工厂类，并遵循开闭原则，即对修改关闭，对扩展开放。
  5. 不滥用工厂模式：不是所有情况下都需要使用工厂模式。对于简单的对象创建，直接使用构造函数可能更简单和直观。
  6. 适当的命名：工厂模式中的类名和方法名应该能够清晰地表达其用途，遵循良好的命名约定。
  7. 不要过度设计：只有在确实需要灵活的对象创建逻辑、需要将对象的创建与使用代码分离时，才应使用工厂模式。不要过度设计，避免增加不必要的复杂性。
  8. 使用单例模式慎重：在工厂模式中使用单例模式可能导致工厂本身成为全局状态，增加了代码的复杂性和难以测试性。除非明确需要，否则慎用单例模式。

解析

工厂模式与模板模式类似但是他们的关注点是不同的

  • 模板模式主要用于定义算法骨架，在一组相关类中提取共同行为，将这些行为放在一个抽象类中，并允许子类通过实现特定方法来完成算法的具体步骤
  • 工厂模式主要用于创建对象，客户端通过接口，而不是直接使用构造函数创建对象

但他们都关注于封装和抽象，都是封装某种特定逻辑实现逻辑重用

在这个模式中，主要有四个角色

  • Product（产品）

    产品角色属于框架一方，它定义了在工厂模式中生成的实例应该具有什么样的行为，但是具体的行为则由实现它的子类去定义

  • Creator（创建者）

    创建者属于框架这一方，它是负责生成Prodect角色的抽象类，但具体的处理则由子类角色进行实现，封装了创建产品的主要逻辑，他对实际生成的产品一无所知，他唯一知道的就是创建产品的过程和方法，不用new 关键字生成实例，而是调用生成实例的专用方法来生成实例，这样就可以防止父类和其他具体类耦合

  • ConcreteProduct（具体产品）

    属于具体加工这一方，决定了具体产品

  • ConcreteCreator（具体创建者）

    负责生成具体产品

在这个模式中，如果我们要生成另外一个新的产品，只需创建一个新的包进行编写即可，无需对framework包进行任何修改，即不需要将另一个包引入到框架之中，也就是工厂包不依赖于产品包

Prototype模式

原型模式是一种创建型模式，这种模式是实现了一个原型接口，该接口用于创建当前对象的克隆。当直接创建对象的代价比较大时，则采用这种模式。例如，一个对象需要在一个高代价的数据库操作之后被创建。我们可以缓存该对象，在下一个请求时返回它的克隆，在需要的时候更新数据库，以此来减少数据库的调用

根据实例来生成实例，而不是根据类来生成实例

介绍

意图：用原型实例指定创建对象的种类，并通过拷贝（有时候拷贝会比原本创建这些原型的方式快很多）这些原型创建新的对象。

主要解决：

  1. 对象种类繁多，无法整合到一个类，必须编写很多个类文件
  2. 难以根据类生成实例，生成过程太过复杂，需要很多步骤跟权限
  3. 想解耦框架与生成的实例。这时，不能指定类名来生成实例，而要先注册（注册表的工作交给其他对象，我们只需关心框架的设计）一个原型实例，再通过复制实例来生成新的实例

何时使用：

  1. 当一个系统 应该独立于它的产品创建，构成和表示时。
  2. 当要实例化的类是在运行时刻指定时，例如，通过动态装载
  3. 为了避免创建一个产品类层次平行的工厂类层次时。
  4. 当一个类的实例只能有几个不同状态组合中的一种时。建立响应数目的原型并克隆他们可能比每次用合适的状态手工实例化该类更方便一些。

缺点：

  1. 装备克隆方法需要对该类的功能进行通盘考虑，这对于全新的类不是很难，但对于已有的类不一定很容易，特别当一个类引用不支持串行化的间接对象，或者引用含有循环结构的时候
