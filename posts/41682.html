设计模式

UML 入门

  1. 关联关系：

    • 双向关联：客户与商品（我中有你，你中有我）

    • 单向关联：包含（我包含你）

    • 自关联：链表的结点

    • 多重性关联：表示关联对象在数量上的对应关系

    • 聚合关系：表示整体与部分的关系，但是部分可以脱离整体而存在（你是我的装备）

    • 组合关系：表示整体与部分的关系，但是整体对象可以控制成员对象的生命周期，同生共死（你是我的牙齿）

  2. 依赖关系（我会用到你）

    是一种使用关系，表示一个事物使用另一个事物，通常有三种实现方式

    • 直接把一个对象作为一个类中方法的参数
    • 在一个类的方法中将另一个对象作为其局部变量
    • 在一个类的方法中调用另一个类的静态方法

  3. 泛化关系（继承）

    继承关系，用于表示父类与子类之间的关系，弗雷有被称作基类或超累，子类又被称为派生类。

  4. 接口与实现关系（实现）

面向对象设计原则概述

大多数情况下，开闭原则是目标，里氏代换是基础，依赖倒转是手段，相辅相成

  1. 单一职责原则：一个类只负责一个功能领域中的相应职责，就一个类而言，应该只有一个引起他变化的原因

    理由：如果一个类承担的职责越多（链接数据库，查询信息，创建图标），那么它被复用的可能性就越小

  2. 开闭原则：一个软件实体应当对扩展开放，对修改关闭。软件实体应该尽量在不修改原有代码的情况下进行扩展

    理由：这是可复用设计的一块基石

    关键：抽象化，将系统定义为一个相对稳定的抽象层，将不同的实现行为移至具体的实现层中完成（定义系统的抽象层，再通过具体类来进行扩展，如果需要修改系统行为，无需对抽象层进行任何改动，只需要增加新的具体类来实现新的业务功能即可）

  3. 里氏代换原则：所有引用父类的地方，必须能够透明的使用其子类的对象

    理解：将一个父类对象替换成它的子类对象，程序将不会产生任何错误和异常，反过来则不成立

    理由：历史代换原则是实现开闭原则的重要方式之一（在程序中使用基类类型来对对象进行定义，在运行时再确定其子类类型，使用子类对象来替换父类对象，可以很方便的对系统进行升级）

    关键：将父类设计为抽象类或接口，让子类继承父类或实现父类接口，并实现在父类中声明的方法

  4. 依赖倒转原则：抽象不应该依赖于细节，细节应当依赖于抽象，这对接口编程，而不是针对实现编程

    理由：程序中使用抽象层进行编程，将具体类写在配置文件中，如果系统行为发生变化，只需要对抽象层进行扩展，并修改配置文件即可

    关键：

    • 在程序代码中传递参数或在关联关系中，尽量引用层次高的抽象层类
    • 依赖注入

    示例：

    增加新数据类型需要修改 CustomerDAO

    增加新数据类型只需修改配置文件

  5. 接口隔离原则：使用多个专门的接口，而不使用单一的总接口，客户端不应该依赖那些它不需要的接口

    示例：

    接口承担太多职责，倒是实现类庞大，不同实现类中不得不实现接口中定义的所有方法，灵活性差

  6. 合成复用原则 ：尽量使用对象组合，而不是继承来达到复用的目的

    理由：可以降低类之间的耦合度，一个类变化对其他类造成的影响较少，如果要继承，严格遵循里氏代换

    关键：尽量使用关联关系，少用继承

    PS：合适用复用，何时用继承，has 用复用，is 用继承

    示例：

    如果此时我们要更改数据库，则要修改两个类中的其中一个，违反开闭原则

    新的设计方式中，如果要更改数据库，则可以直接使用依赖注入的方式

  7. 迪米特法则：一个软件实体应当尽可能地减少与其他实体发生相互作用

    理由：如果符合此法则，当其中一个模块发生修改时，就会尽量少地影响其他模块，扩展会相对容易，这是对软件实体之间通信的限制

    关键：尽量创建松耦合的类

    示例：

    在这个界面中，增加新的界面控件需要修改与之交互的其他空间的源代码，导致扩展性差

    引入一个专门控制界面空间交互的中间类，降低界面之间的耦合度，将请求转发给中间类，再又中间类来完成对其他控件的调用

创建型模式

创建型模式将对象的创建和使用分离，在使用对象时无需关心对象的创建细节，从而降低系统的耦合度。关键：创建什么，由谁创建，何时创建

Singleton

确保一个类只有一个实例，并且自行实例化并向系统提供这个实例，这个类成为单例类，提供全局访问方法

这种模式涉及到一个单一的类，该类负责创建自己的对象同时确保只有单个对象被创建。这个类提供了一种访问其唯一对象的机制，可以直接访问，不需要实例化该类的对象

介绍

动机：保证一个类仅有一个实例，并提供一个访问它的全局访问点

  • 确保对象只有一个实例
  • 想在程序上表现出“只存在一个实例”

主要解决：一个全局使用的类频繁的创建与销毁

何时使用：

  • 系统只需要一个实例对象，例如系统要求提供一个唯一序列号生成器或资源管理器，或者需要考虑资源消耗太大而只允许创建一个对象
  • 客户调用类的单个实例只允许使用一个公共访问点，除了该公共访问点，不能通过其他途径访问该实例

如何解决：判断系统是否已经有这个单例，如果有则返回，如果没有则创建。

关键代码：私有构造函数，设置私有静态成员变量，增加共有静态访问方法

应用实例：

  • Windows 是多进程多线程的，在操作一个文件的时候，不可避免的会出现多个进程或线程同时操作一个文件的现象，所以所有文件的处理必须通过唯一的实例来进行

  • 一些设备管理器常常设计为单例模式，比如一个电脑有两台打印机，在输出的时候就要处理不能两台打印机打印同一个文件

  • 负载均衡器的设计

  • 生产唯一序列号，计数器等

优点：

  • 在内存里只有一个实例，减少了内存的开销，避免频繁的创建和销毁实例
  • 提供了对唯一实例的受控访问，封装了唯一实例，可以严格控制客户怎样以及何时访问
  • 可以通过扩展，让单例 变为 n 例

缺点：

  • 没有抽象层，扩展困难
  • 职责过重，违背单一职责原则。既提供了业务方法，又提供了创建对象的方法，将对象的创建和本身的功能进行耦合
  • 由于自动垃圾回收技术，实例化共享对象长时间不被利用，会被系统认为是垃圾，会自动销毁并回收资源，会导致共享单例对象状态丢失

注意事项：

需要注意多线程问题，记得加同步锁

假设一个单例对象还没有被创建，且创建需要消耗大量时间，但同时又两个请求要访问这个单例对象，那么这个时候有可能会创建出两个对象，违背单例模式的初衷

实现

饿汉式，线程安全，在类加载的时候就已经创建了单例对象，缺点是不能在系统需要的时候才创建这个单例类，可能会造成资源浪费

懒汉式，延迟加载，使用 synchronized 关键字修饰 get 方法，避免多个线程同时调用 get 方法，缺点，每次调用该方法的时候都需要进行线程判定，会导致系统性能降低，不适用于多线程高并发访问环境

改进：锁定临界资源块，问题依然存在，如果两个线程同时调用 get 方法，都能通过判断，导致依然可能产生多个单例对象

改进：使用双重检查锁定（检查，锁定，检查），需要在成员变量中添加 instance 修饰符，确保多个线程都能正确处理，会屏蔽虚拟机所做的一些代码优化

饿汉式单例类不能实现延迟加载，始终占据内存，懒汉式单例类线程安全控制繁琐，且受性能影响

Initialization on Demand Holder(IoDH)，将成员变量改为 静态内部类 在内部类中创建单例对象，再将该单例对象通过 get 方法返回给外部使用

单例对象没有作为成员变量直接实例化，因此类加载时不会实例化

Simple Factory

定义一个工厂类，可以根据参数的不同方位不同的类实例，被创建的实例通常都具有共同的父类。因为简单工厂模式中用于创建实例的方法是静态的方法，因此简单工厂模式又被称为静态工厂方法模式

介绍

动机：将类的创建和使用进行解耦

主要解决：单一类职责过重，创建和使用都在一个类中

何时使用：

  • 工厂类负责创建的对象比较少，不会造成工厂方法业务逻辑负责
  • 客户端只知道传入工厂类的参数，对于如何创建对象并不关心

如何解决：工厂（核心，根据传入的参数创建不同的产品），抽象产品（提供所有产品的公共业务方法和抽象业务声明方法），具体产品（实现抽象产品）

关键代码：工厂类提供静态方法，该方法根据传入的参数来创建不同的产品

优点：

  • 工厂类包含必要判断逻辑，可以决定什么时候创建哪个产品类的实例，客户端可以免除直接创建产品对象的职责，而仅仅只是消费产品（客户端只负责消费），实现类对象创建和使用的分离
  • 客户端无需知道创建具体产品的类名，只需要知道具体产品所对应的参数即可
  • 通过引入配置文件，可以在不修改任何客户端代码的情况下更换和增加新的具体产品类，提高系统灵活性

缺点：

  • 工厂类集中了所有产品的创建逻辑，职责过重，一旦不能正常工作，整个系统都将受到影响
  • 使用简单工厂模式会增加系统中类的个数，增加系统的复杂度和理解难度
  • 系统扩展困难，一旦添加新产品就得修改工厂逻辑，不利于系统的扩展和维护
  • 使用静态工厂方法，工厂角色无法形成基于继承的等级结构

注意事项：

  • 不适用于存在大量产品的情况

Factory Method

定义一个用于创建对象的接口，让子类决定将哪一个类实例化

介绍

动机：在静态工厂模式中，静态工厂方法根据传入参数的不同来创建不同的产品，如果增加新产品，那么就一定会修改原代码

主要解决：简单工厂方法优化，解决工厂类职责太重的问题

何时使用：

  • 不想让客户端知道所需对象的具体类
  • 需要一个容易扩展的系统可以考虑使用

如何解决：抽象产品，具体产品，抽象工厂，具体工厂

关键代码：在简单工厂模式的基础上，抽象工厂，让具体工厂生产具体产品

应用实例：

  • 日志记录器，根据不同的文件来源创建不同的日志记录器
  • 各种开源框架和 API 类库的核心模式

优点：

  • 使用具体工厂方法创建需要的产品，向客户隐藏了哪种具体产品将被实例化这一细节，客户只需要关心所需产品的对应工厂，而无须关心创建细节，甚至不需要知道具体产品类的类名
  • 系统加入新产品时，无需修改抽象工厂和抽象产品提供的接口，无须修改客户端，无需修改其他具体类和具体产品，只需添加一个具体工厂和具体产品就行，完全符合开闭原则

缺点：

  • 增加新产品时，要添加具体工厂和对应的具体产品，系统中类的个数将成对增加
  • 引入抽象层，在客户端代码中均使用抽象层进行定义，增加系统的抽象性和理解难度，实现的时候可能需要用到 DOM ，反射，等技术，增加实现难度

注意事项：每个工厂只生产一类产品，可以考虑将相关产品组成一个产品族，由同一个工厂进行生产（抽象工厂模式）

Abstract Factiory

在工厂方法模式中，由于一种产品由一个工厂生产，这个时候，如果增加新的产品就会导致代码量急速增加，将一些相关产品组成一个产品族，由一个工厂统一生产，会更加便于系统扩展

Prototype

使用原型实例来创建对象的种类，通过克隆一个原型实例来创建对象

介绍

动机：简化相同对象的创建

主要解决：创建一个对象成本很高，但又需要创建多个这样的相似对象，可以使用原型模式简化对象的创建过程

何时使用：

  • 创建新对象的成本较大（初始化占用时间长，占太多的CPU资源或网络资源）新对象可以通过原型模式对已有对象来复制获得，如果是相似对象，则可以对成员变量稍作修改
  • 如果系统要保存对象的状态，而对象的状态变化很小，或者对象本身占用的内存较少时，可以使用原型模式配合备忘录模式
  • 需要避免使用分层次的工厂类来创建分层次的对象，且类的实例对象只有一个或很少的几个组合状态，通过复制原型对象得到新实例可能比使用构造函数创建一个新实例更方便

如何解决：抽象原型类（声明克隆方法的接口），具体原型类（实现抽象原型类中声明的克隆方法，在克隆方法中返回自己的一个克隆对象）

关键代码：实现克隆接口

应用实例：

优点：

  • 简化对象创建过程，通过复制一个已有实例可以提高实例的创建效率
  • 扩展性好，在原型模式中提供了抽象原型类，在客户端可以针对抽象原型类进行编程，将具体原型类写在配置文件中，增加或减少对系统没有任何影响
  • 提供简化的创建接口，无需专门的工厂类来创建产品
  • 可以使用深克隆的方式保存对象的状态。

缺点：

  • 需要为每一个类配备一个克隆方法，且克隆方法位于一个类的内部，当要对该类进行修改的时候，需要修改源代码，违反开闭原则
  • 在实现深克隆的时候需要编写大量复杂的代码

注意事项：

Builder

介绍

动机：将对象的创建进行隔离

主要解决：复杂对象的组装与创建

何时使用：

  • 需要生成的产品对象有复杂的内部结构，这些产品对象通常包含多个成员变量
  • 需要生成的产品对象属性相互依赖，需要指定其生成顺序
  • 隔离复杂对象的创建和使用
  • 需要让对象的创建过程独立于创建该对象（创建对象的方法封装在指挥者类中）

如何解决：

  • 抽象建造者，（声明两类方法，一种是建造产品部件的方法，一种是直接返回具体产品的方法）
  • 具体建造者（实现抽象建造者），
  • 产品角色（被构建的复杂对象，包含多个组成部件，具体建造者创建产品内部表示并定义其装配过程），
  • 指挥者（安排复杂对象的建造次序，指挥者与抽象建造者之间存在关联关系，可以在其 construct 建造方法中调用建造者对象的不叫构造与装配方法，完成复杂对象的建造，客户端一般只需要与指挥者进行交互），隔离客户与创建过程，针对抽象构造者编程

关键代码：指挥者和建造者的编写（类似抽象工厂模式，不过抽象工厂模式生成的是一个产品等级，一组产品，建造者模式生成的是一个由多个产品组成的复杂产品）

应用实例：

优点：

  • 客户端不必知道产品内部的组成细节，将创建产品与产品本身进行结构使得相同的创建过程可以创建不同产品
  • 每一个具体建造者都相对独立，而与其他具体建造者无关，可以很方便地替换具体建造者或增加具体建造者
  • 增加指挥者类针对抽象建造者编程，使得增加新的建造者无须修改原有代码，扩展方便，符合开闭原则
  • 更加精细的控制产品的创建过程，将复杂产品的创建分解在不同的方法中，使得创建过程更加清晰

  • 创建的产品一般具有较多共同点，组成部分相似，如果差异性很大，不适用（适用范围有限制）

将请求的处理者组织成一条链，并让请求沿着链传递，由链上的处理者对请求进行相应的处理，客户端无须关心请求的处理细节以及请求的传递。

避免将请求发送者与接收者耦合在一起，让多个对象都有机会接收请求，将这些对象链接成一条链，并沿着这条链传递请求

  • 避免多重条件选择语句
  • 提供了一种算法复用机制

缺点：

  • 客户端必须知道所有的策略类，并且自行决定使用哪一个策略类
  • 可能会使系统产生很多的具体策略类
  • 无法在客户端同时使用多个策略类

Visitor

提供一个作用于某对象结构中的各元素的操作表示，它使得可以在不改变各元素的类的前提下定义作用于这些元素的新操作

介绍

动机：操作复杂对象结构

主要解决：

何时使用：

  • 一个对象结构包含多个类型的对象，希望对这些对象实施一些依赖其具体类型的操作。在访问者中针对每一种具体的类型都提供一个访问操作，不同类型对象可以有不同的访问操作
  • 需要对一个对象结构中的对象进行很多不同且不相关的操作，且需要避免让这些操作污染这些对象的类，也不希望在增加新操作时修改这些类。
  • 对象结构中的对象对应的类很少改变，但经常需要在此对象结构上定义新的操作

如何解决：

  • 抽象访问者：为对象结构中每一个具体元素类声明一个访问操作，从这个操作名称或参数类型可以清楚的知道需要访问的具体元素的类型，具体访问者需要实现这些操作的方法
  • 具体访问者：实现了每个由抽象访问者声明的操作，每一个操作用于访问对象结构中的一种类型元素
  • 抽象元素：抽象元素一般是抽象类或者接口，它定义一个 accept 方法，该方法通常以一个抽象访问者作为参数
  • 具体元素：具体元素实现了 accept 方法，在 accept 方法中调用访问者的访问方法以便完成对一个元素的操作
  • 对象结构：对象结构是一个元素的集合，它用于存放元素对象，并且提供了遍历其内部元素的方法。它可以结合组合模式来实现
